This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A museum exhibit about targeted drug delivery. Visitors build a "nanoparticle" by placing colored ligand pieces into 6 slots on a physical board and tap an RFID-tagged "puzzle" card. The system reads sensors, then 1–4 simulation PCs run a p5.js fluid/cell simulation showing how well the nanoparticle binds the puzzle's tumor versus collateral tissues (heart, liver, lung).

Three execution modes coexist:
- **Local dev (browser-only):** open `concept_development/simulation_prototype/index.html`. `dashboard.html` drives it via `BroadcastChannel` (same browser only).
- **Exhibit (Pi + clients):** `pi/master_server.py` is a Flask + Socket.IO server. Sim PCs load `index.html?server=<pi-ip>:5000` and become Socket.IO clients. The Pi also serves a results display at `/` and an admin page at `/admin`.
- **GitHub Pages:** `.github/workflows/deploy-pages.yml` publishes `concept_development/simulation_prototype/` so the simulation is browsable online (no hardware, no Pi).

`java_prototype/` is an old Processing sketch retained for reference. The simulation prototype is the active code.

## Architecture — the cross-system flow

Understanding this codebase requires holding three runtimes at once: the Arduino firmware, the Pi server, and the browser simulation.

### Hardware → Pi
- **Color sensors:** 6× APDS-9960 sit on a TCA9548A I2C multiplexer (all share I2C address 0x39). [pi/sensor_service.py](pi/sensor_service.py) reads each channel and maps RGB → color name using [pi/color_map.json](pi/color_map.json) (regenerated by [color_calibration.py](pi/color_calibration.py)). I2C reads are blocking and slow (~1 s for a full sweep due to the configured 64-cycle integration time).
- **RFID + button:** Arduino Uno with the Adafruit PN532 shield runs [arduino/tdd_rfid/tdd_rfid.ino](arduino/tdd_rfid/tdd_rfid.ino), emits one-line JSON over USB serial (`{"type":"tag", ...}`, `{"type":"tag_removed"}`, `{"type":"button", ...}`). [pi/arduino_rfid_service.py](pi/arduino_rfid_service.py) reads the serial port on a background OS thread.

### Pi server internals — the eventlet/OS-thread boundary
[pi/master_server.py](pi/master_server.py) uses `flask_socketio` with `async_mode="eventlet"`. **`socketio.emit` from a plain OS thread is unreliable for reaching remote clients.** Two patterns address this:

1. **Arduino callbacks → `_action_queue` → `_action_dispatch_loop` greenlet.** Anything triggered by the Arduino reader thread is enqueued and a long-running eventlet greenlet picks it up so emits happen inside the event loop.
2. **Blocking I2C reads → `eventlet.tpool.execute(...)`.** Sensor reads run in a real OS thread; the calling greenlet yields cooperatively.

Two more long-running greenlets:
- `_sensor_poll_loop` — sweeps sensors ~once/second, only when an RFID tag is present (no tag → emit empty state once and idle). During TESTING it pushes live readings to the display but does **not** mutate `state_machine` (so a brief lift of the nanoparticle won't corrupt the test snapshot).
- `_test_watchdog_loop` — auto-resets if a TESTING state runs longer than `TEST_TIMEOUT_SECONDS`.

### Pi state machine
[pi/state_machine.py](pi/state_machine.py): `IDLE → NANOPARTICLE_SCANNED → PUZZLE_LOADED → TESTING → RESULTS → IDLE`. Two non-obvious behaviors:
- **Pending puzzle queue:** a tag scanned during TESTING is stored in `pending_puzzle` and applied when the test ends (via `restart_test()` or `complete_test()`). This handles chaotic visitor tag-swapping without dropping the swap.
- **Restart preserves state:** `action_restart_test` (start button mid-test) emits `reset` to clients, steps state back to PUZZLE_LOADED keeping `current_puzzle` and `ligand_positions`, then calls `action_start_test` — no re-scan needed. A 2-second debounce in `_BUTTON_DEBOUNCE_SECS` blocks accidental double-presses.

### Pi → simulation clients
[pi/client_manager.py](pi/client_manager.py) tracks Socket.IO clients and assigns tissues (4 tissues split across 1–4 clients via `t % n == i`). At test start the manager **locks**: the set of "expected completers" is snapshotted so mid-test joiners become observers without tissue assignments and mid-test disconnects don't block completion. `no_completers_left()` triggers an auto-reset if everyone vanishes.

### Simulation client (browser)
- Entry: [src/main.js](concept_development/simulation_prototype/src/main.js). Loads `puzzle_example.json` (local-dev fallback), creates `Simulation` instances (one per assigned tissue, or all 4 in 2×2 grid), then sets up the network — **in that order**, so any `start_test` arriving immediately after `register_client` finds simulations ready.
- [src/network.js](concept_development/simulation_prototype/src/network.js): if URL has `?server=host:port`, connect via Socket.IO; otherwise fall back to `BroadcastChannel("tdd-channel")` for local dashboard dev.
- `?tissue=N` (0–3) renders one tissue full-screen instead of a 2×2 grid; this is how each exhibit sim PC is launched.
- `?fluid=true` enables the WebGL2 `FluidSimulation` (Stam-style stable fluids). Otherwise particles use simple noise-based turbulence.
- Each `Simulation` uses a soft-body spring-mass cell membrane ([Cell.js](concept_development/simulation_prototype/src/Cell.js)). Particles bind cells via leading-edge ligand/receptor adjacency matching ([BindingLogic.js](concept_development/simulation_prototype/src/BindingLogic.js)). Tunable physics/visual constants live in [src/constants.js](concept_development/simulation_prototype/src/constants.js).

### Puzzle JSON format
On disk (in [pi/puzzles/](pi/puzzles/)) puzzles use named entries for human readability:

```json
"receptors": [{"color": "Red", "value": 0.8}, ...]
"ligands":   [{"color": "Red", "count": 2}, ...]
```

The simulation consumes 6-element indexed arrays. Both sides convert at the JSON boundary and **must stay in sync**:
- JS: `normalizePuzzle()` in [src/utils.js](concept_development/simulation_prototype/src/utils.js)
- Py: `normalize_puzzle()` in [pi/puzzle_format.py](pi/puzzle_format.py)

Color order is fixed: `["Red", "Blue", "Green", "Purple", "Orange", "Yellow"]` — defined in `LIGAND_COLORS` in [pi/config.py](pi/config.py) and `COLOR_NAMES` in [src/constants.js](concept_development/simulation_prototype/src/constants.js).

Tag → puzzle mapping lives in [pi/puzzles/index.json](pi/puzzles/index.json) and is editable live via `/admin`. The Arduino service reloads the index when the admin endpoints save changes — no server restart needed.

## Robustness — the exhibit runs unattended in a chaotic environment

This is a museum exhibit. Visitors (often kids) will mash the start button repeatedly, swap RFID tags mid-test, yank the nanoparticle off the sensors and slam it back, plug/unplug ligand pieces during a run, and walk away mid-interaction. There is no operator nearby. **Any code path that changes state must assume hostile input timing.** A wedged exhibit means staff has to power-cycle the Pi, so "fail open / auto-recover" beats "fail correct" every time.

Existing patterns that exist *because* of this and must be preserved when extending the system:

- **Debounce on the start button** (`_BUTTON_DEBOUNCE_SECS` in `master_server.py`) — double-presses must not launch two sequential tests.
- **Pending puzzle queue** in `state_machine.py` — a tag scanned during TESTING is queued, not dropped or applied immediately.
- **Client-manager lock** in `client_manager.py` — the set of expected completers is snapshotted at test start so mid-test joiners/disconnects don't block completion.
- **Test watchdog** (`_test_watchdog_loop`) — auto-resets if a test exceeds `TEST_TIMEOUT_SECONDS`, covering stalled clients, network failures, or any unforeseen wedge.
- **`no_completers_left()` auto-reset** — if every expected sim client disconnects mid-test, the exhibit resets rather than hanging.
- **Sensor poll skips state mutation during TESTING** — momentarily lifting the nanoparticle won't corrupt the running test's snapshot.
- **Tag-present gating in `_sensor_poll_loop`** — empty sensor reads don't churn state when no nanoparticle is on the pad.
- **Defensive guard on `start_test` in the browser** ([main.js](concept_development/simulation_prototype/src/main.js)) — duplicate or stale `start_test` events while already in test mode are ignored.
- **Socket.IO `reconnectionAttempts: Infinity`** — sim PCs that boot before the Pi recover silently once the Pi is up.
- **`uid` passed explicitly to `action_scan_rfid`** — avoids a race where `current_tag_uid` is cleared by a `tag_removed` event before the lookup runs.

When adding state-changing code, work through the failure modes explicitly:
- What if this fires twice in 100 ms?
- What if the user removes the trigger (tag, ligand, button) mid-operation?
- What if a sim client disconnects right now?
- If this can wedge the exhibit, is there a watchdog or auto-reset that will recover it?

Prefer queue-and-drain (like `pending_puzzle`), snapshot-at-start (like the completer lock), and time-bounded watchdogs over assuming inputs arrive in a sane order.

## When making changes

- **Touching `master_server.py`?** Decide whether the new code runs in a greenlet or an OS thread. If it must emit to remote clients from an OS thread, route it through `_action_queue`. If it makes a blocking call (I2C, file IO that could stall), wrap it in `eventlet.tpool.execute`.
- **Adding a state transition?** Update both `TRANSITIONS` in [pi/state_machine.py](pi/state_machine.py) and the corresponding `action_*` orchestration in [pi/master_server.py](pi/master_server.py). Consider the pending-puzzle and locked-clients paths.
- **Changing puzzle JSON shape?** Update `normalizePuzzle` in JS *and* `normalize_puzzle` in Python, plus the example file [puzzle_example.json](concept_development/simulation_prototype/puzzle_example.json) and the puzzles in [pi/puzzles/](pi/puzzles/).
- **Editing simulation JS?** `index.html` lists scripts in dependency order (constants → utils → scoring → sprites → physics → Simulation → network → main). Add new files there. Locally just refresh the browser. In exhibit mode the Pi serves the same files via `/sim/<path>`, so new files become available without redeploy.
- **Editing the simulation for the GitHub Pages site?** It deploys `concept_development/simulation_prototype/` as-is; nothing exhibit-specific (Socket.IO, Pi server) runs there because there's no `?server=` param.
- **Hardware changes** (sensor count, multiplexer address, integration time, color list) live in [pi/config.py](pi/config.py). Re-run [color_calibration.py](pi/color_calibration.py) after physical changes to ligand pieces, mounting, or lighting.
